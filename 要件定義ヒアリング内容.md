### RPA開発 要件定義ヒアリング項目リスト

大きく分けて7つのカテゴリでヒアリングを進めると、情報を整理しやすくなります。

1.  **業務の概要と目的のヒアリング**
2.  **現状業務（As-Is）のヒアリング**
3.  **RPA化の要件（To-Be）の定義**
4.  **対象システム・データに関するヒアリング**
5.  **例外処理・エラーハンドリング**
6.  **非機能要件に関するヒアリング**
7.  **運用・保守に関するヒアリング**

---

### 1. 業務の概要と目的のヒアリング

**【目的】**
プロジェクトの全体像を把握し、RPA化が目指すべきゴールを明確にします。これにより、開発の優先順位付けや費用対効果の判断がしやすくなります。

| ヒアリング項目 | 具体的な質問例・確認例 |
| :--- | :--- |
| **業務の名称** | ・この業務の正式な名称は何ですか？（例：「月次売上実績報告書作成業務」）<br>・社内では普段、何と呼ばれていますか？ |
| **担当部署・担当者** | ・主担当の部署とご担当者様のお名前を教えてください。<br>・この業務に関わる他の部署や担当者はいますか？ |
| **RPA化の目的・ゴール** | ・なぜこの業務をRPA化したいのですか？（**最重要質問**）<br>　・例：「作業時間を月50時間削減したい」<br>　・例：「入力ミスをゼロにして、手戻りをなくしたい」<br>　・例：「担当者の精神的負担を軽減し、より付加価値の高い業務に集中させたい」 |
| **業務の発生頻度・タイミング** | ・この業務はどのくらいの頻度で発生しますか？（毎日、毎週月曜、毎月25日、不定期など）<br>・1日のうち、どの時間帯に作業することが多いですか？（例：「毎朝9時にメールが届いたら開始」） |
| **業務の成果物** | ・この業務が完了したときの最終的なアウトプットは何ですか？<br>　・例：「Excelの報告書が作成され、特定の共有フォルダに格納される」<br>　・例：「関係者宛に完了報告メールが送信される」 |

---

### 2. 現状業務（As-Is）のヒアリング

**【目的】**
現在の業務手順をミクロの視点で完全に理解します。**「百聞は一見に如かず」**です。必ず担当者に**実際のPC操作を画面共有してもらいながら**、一つ一つの手順を確認してください。

| ヒアリング項目 | 具体的な質問例・確認例 |
| :--- | :--- |
| **業務全体の流れ（デモ）** | ・**実際に業務を行っている画面を見せていただきながら、最初から最後までの手順を説明していただけますか？**<br>・（開発者側で）画面を録画してもよろしいでしょうか？ |
| **業務の開始トリガー** | ・何をもってこの業務を開始しますか？<br>　・例：「部長から『〇〇作成依頼』という件名のメールが届いたら」<br>　・例：「毎朝10時になったら」 |
| **具体的な操作手順（ステップ・バイ・ステップ）** | ・**（操作を見ながら）**<br>　・そのファイルはどこから開いていますか？<br>　・そのボタンをクリックする判断基準は何ですか？<br>　・その項目には何を入力していますか？転記元はどこですか？<br>　・複数のパターンがある場合、どのように判断して処理を分けていますか？（If文の条件分岐） |
| **処理時間と件数** | ・この業務を1件処理するのに、平均で何分かかりますか？<br>・1日あたり（または1ヶ月あたり）に何件くらい処理しますか？<br>・繁忙期などはありますか？その際の件数はどのくらいですか？ |
| **判断が必要な箇所** | ・業務の中で、担当者の方が「考えて」判断している箇所はありますか？<br>　・例：「この申請者の過去の実績を見て、承認するか決める」→ **ルール化できない判断はRPA化が難しい**ため、特に注意してヒアリングする。 |

---

### 3. RPA化の要件（To-Be）の定義

**【目的】**
現状業務を踏まえ、「ロボットに何をさせるか」を具体的に定義します。曖昧な部分をなくし、開発者とクライアントの認識を完全に一致させます。

| ヒアリング項目 | 具体的な質問例・確認例 |
| :--- | :--- |
| **ロボットの処理範囲** | ・現状業務のどこからどこまでをロボットに任せますか？<br>・ロボットが処理できない部分は、人間が引き継ぐ想定ですか？その連携方法は？<br>　・例：「データのダウンロードと整形までをロボットが行い、最終的な分析と考察は人間が行う」 |
| **インプットデータ** | ・ロボットが処理を開始する際に必要なデータ（ファイル、メールなど）はどこにありますか？<br>・ファイル名やフォルダ構成にルールはありますか？（例：`YYYYMMDD_売上データ.csv`）<br>・インプットデータのフォーマットは常に同じですか？ |
| **処理ルール（ビジネスロジック）** | ・データをどのように加工・計算しますか？具体的なルールを教えてください。<br>　・例：「A列の値が『東京』の場合、B列に『1』をセットする」<br>　・例：「請求金額が100万円以上の場合は、ステータスを『要承認』にする」 |
| **アウトプット** | ・処理結果をどのような形式で、どこに出力しますか？<br>・ファイル名はどのように命名しますか？（例：`【処理済】YYYYMMDD_報告書.xlsx`）<br>・処理完了後、誰にどのような内容で通知（メール、Teamsなど）しますか？ |
| **正常終了時の後処理** | ・処理が正常に完了した後、インプットファイルはどうしますか？<br>　・例：「『処理済みフォルダ』に移動させる」「削除する」 |

### 4. 対象システム・データに関するヒアリング

**【目的】**
ロボットが操作するアプリケーションや環境の技術的な詳細を確認します。セレクターの安定性や実行環境の差異によるエラーを未然に防ぎます。

| ヒアリング項目 | 具体的な質問例・確認例 |
| :--- | :--- |
| **利用アプリケーション** | ・業務で利用する全てのアプリケーション、Webサイト（URL）、フォルダパスをリストアップしてください。<br>　・例：SAP、Salesforce、自社開発のWebシステム、Excel、Outlook、共有フォルダなど |
| **実行環境** | ・ロボットを実行するPC（またはサーバー）のOS、画面解像度、ブラウザの種類とバージョンを教えてください。<br>・開発環境と本番環境に違いはありますか？ |
| **ログイン情報** | ・各システムへのログインIDとパスワードはどうしますか？<br>　・**（注意）** パスワードそのものを聞くのではなく、**「ロボット専用のアカウントを用意するか」「OrchestratorのAssetsで管理するか」**といった管理方法を確認します。 |
| **アプリケーションの挙動** | ・アプリケーションのレスポンスが遅い（表示に時間がかかる）ことはありますか？<br>・時間帯によって重くなることはありますか？ |

### 5. 例外処理・エラーハンドリング

**【目的】**
予期せぬ事態が発生した際にロボットが停止せず、適切に対応できるように設計します。堅牢なロボットを作るための最重要パートの一つです。

| ヒアリング項目 | 具体的な質問例・確認例 |
| :--- | :--- |
| **業務上の例外** | ・これまでの業務で、通常とは異なるパターンが発生したことはありますか？<br>　・例：「インプットファイルに必要な列が存在しなかった」<br>　・例：「本来数字が入るべきセルに、文字列が入っていた」<br>　・例：「マスタデータに存在しない顧客コードがあった」<br>・**その場合、現在はどのように対応していますか？**（→これをロボットの例外処理のルールにする） |
| **システム的なエラー** | ・もし、対象のシステムがフリーズしたり、ネットワークが切断されたりした場合はどうしますか？<br>　・例：「3回までリトライし、それでもダメなら処理を中断して管理者にメール通知する」 |
| **エラー時の対応** | ・エラーが発生した場合、ロボットは処理を中断すべきですか？それとも、そのデータだけをスキップして次の処理を続けるべきですか？<br>・エラーが発生した際、誰に、どのような情報（エラーメッセージ、スクリーンショットなど）を通知すべきですか？ |

### 6. 非機能要件に関するヒアリング

**【目的】**
性能、セキュリティ、スケジュールなど、機能以外の要件を定義します。

| ヒアリング項目 | 具体的な質問例・確認例 |
| :--- | :--- |
| **性能要件** | ・ロボットの処理速度に目標はありますか？（例：「1時間あたり100件以上処理してほしい」）<br>・全件処理するのに、最大でどれくらいの時間を想定していますか？ |
| **スケジュール要件** | ・（Orchestrator利用の場合）ロボットをいつ実行したいですか？（例：「平日の毎朝6時に実行」）<br>・同時に複数のロボットが動く可能性はありますか？ |
| **セキュリティ要件** | ・個人情報や機密情報を扱いますか？<br>・パスワードなどの機密情報はどのように管理しますか？（Orchestrator Assetsの利用を推奨） |

---

### 7. 運用・保守に関するヒアリング

**【目的】**
開発完了後、ロボットを安定して動かし続けるための体制やルールを決めます。

| ヒアリング項目 | 具体的な質問例・確認例 |
| :--- | :--- |
| **運用体制** | ・ロボットの実行監視は誰が行いますか？<br>・エラーが発生した際の一次対応は、どの部署の誰が担当しますか？ |
| **処理結果のレポーティング** | ・ロボットの実行結果（処理件数、成功件数、失敗件数など）を誰に報告する必要がありますか？<br>・報告の形式（メール、ログファイルなど）に希望はありますか？ |
| **仕様変更時の対応** | ・将来的に、この業務の手順や使用するシステムが変更される予定はありますか？<br>・変更があった場合、誰が開発者（または保守担当）に連絡しますか？ |

---

### ヒアリングを成功させるための追加のヒント

*   **業務手順書（PDD）を作成する:** ヒアリングした内容をドキュメントにまとめ、クライアントに提示して「この内容で間違いありませんか？」と**署名（サインオフ）**をもらいましょう。これにより、後の「言った・言わない」問題を防げます。
*   **専門用語を避ける:** クライアントはRPAの専門家ではありません。「セレクター」や「アセット」といった言葉は避け、「どのボタンですか」「パスワードの管理方法はどうしますか」のように平易な言葉で質問しましょう。
*   **理想論だけでなく現実を確認する:** 「本来はこうあるべき」という話と、「実際にはイレギュラーなデータがよく来る」という現実の両方を聞き出すことが重要です。
